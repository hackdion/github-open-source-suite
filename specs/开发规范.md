# 开发规范

本文档定义“开源到 GitHub 的标准流程技能”项目的文档风格、命名约定、Git 工作流和 AI 协作协议，确保仓库长期可维护。

---

## 1. 语言与运行时要求

| 维度 | 规格 |
|------|------|
| **语言** | Markdown、YAML、少量 Python 辅助脚本 |
| **运行时** | `python3`、`git`、`gh`、`rg` |
| **包管理器** | 当前仓库级不引入统一包管理器；如必须新增依赖，优先复用外部现有工具 |
| **严格模式** | 不适用传统编译严格模式；文档与技能通过结构校验、占位符扫描、git/gh 回读验证 |

---

## 2. 代码风格与 Lint 规则

### 2.1 工具链

| 工具 | 职责 | 配置文件 |
|------|------|----------|
| **`quick_validate.py`** | 技能 frontmatter 与基础结构校验 | 外部工具：`~/.codex/skills/.system/skill-creator/scripts/quick_validate.py` |
| **`git diff --check`** | 空白符与 patch 级错误检查 | Git 内置 |
| **占位符扫描** | 检查公开文档是否遗留 `{{...}}` / `TBD` | 命令式校验，无单独配置文件 |

### 2.2 Lint 规则概要

当前仓库不引入独立 lint 配置文件，必须使用下列命令作为最小门禁：

```jsonc
{
  "requiredChecks": [
    "python3 ~/.codex/skills/.system/skill-creator/scripts/quick_validate.py <skill-dir>",
    "git diff --check",
    "public-doc placeholder scan",
    "git status --short --branch",
    "gh api / gh release view when GitHub state is claimed"
  ]
}
```

### 2.3 格式化配置

```jsonc
{
  "markdown": {
    "lineBreakStyle": "logical-wrap",
    "fencedCodeBlocks": true,
    "explicitHeadings": true
  },
  "yaml": {
    "frontmatterRequiredForSkills": true,
    "twoSpaceIndent": true
  }
}
```

**关键决策说明**：

- **不引入额外格式化依赖**：当前仓库以文档为主，优先保持低依赖和可移植性。
- **技能结构校验优先于样式工具**：项目风险主要来自前置条件、handoff 与占位符，而不是代码排版。

### 2.4 IDE 配置

```jsonc
{
  "files.trimTrailingWhitespace": true,
  "files.insertFinalNewline": true,
  "editor.wordWrap": "on",
  "editor.formatOnSave": false
}
```

## 4. Markdown / YAML / Python 特定规范

### 4.1 技能 frontmatter 规范

```yaml
# ✅ 正确写法
---
name: oss-repo-audit
description: Audit a local repository for open-source readiness before GitHub publication.
---

# ❌ 错误写法
---
name: OSS Repo Audit
when: open source
---
```

### 4.2 Markdown 结构规范

| 场景 | 偏好 | 原因 |
|------|------|------|
| **项目治理文档** | 明确标题与分节 | 便于后续 Skill 稳定读取 |
| **dogfood 报告** | 固定四类问题结构 | 便于横向比较多仓库反馈 |
| **release 文档** | 中英双语分节 | 匹配当前仓库受众与发布方式 |

### 4.3 避免的写法

```md
<!-- ❌ 避免：把未确认的值直接写进公开文档 -->
Maintainer email: support@example.com

<!-- ✅ 替代方案：写真实值，或明确说明当前不适用 -->
Maintainer contact is routed through GitHub Discussions + Issues.
```

```yaml
# ❌ 避免：frontmatter 中出现未被校验脚本支持的字段
name: oss-release-management
summary: release helper

# ✅ 替代方案
name: oss-release-management
description: Standardize first-release and ongoing release work for an open-source repository.
```

```python
# ❌ 避免：用脚本偷写大量文件内容，绕开仓库审阅
Path("README.md").write_text(generated_text)

# ✅ 替代方案：优先通过补丁显式修改，脚本仅做校验或结构扫描
```

## 5. Skill Suite 特定规范

### 5.1 设计原则

- **单技能单职责**：每个 `oss-*` 目录只解决一个阶段问题。
- **共享规范集中维护**：跨技能规则统一放在 `shared/references/`。
- **模板集中维护**：复用模板统一放在 `shared/assets/`。

### 5.2 技能目录结构

```text
oss-example/
├── SKILL.md
└── agents/
    └── openai.yaml
```

### 5.3 导入导出规则

- 技能目录禁止各自复制一份共享规范
- 技能正文必须引用共享规范，而不是把规则重新抄一遍
- `agents/openai.yaml` 必须和 `SKILL.md` 的触发语义保持一致

### 5.4 技能正文排序

```text
1. frontmatter
2. 标题
3. Overview
4. Workflow
5. Required References / Templates
6. Rules / Preconditions
7. Handoffs / Output Expectations
```

### 5.5 不适用项

- 本仓库没有 React/Vue 等 UI Hook 约束
- 本仓库没有传统模块 import 排序约束作为主规则
- 若后续新增脚本，再为代码目录单独补充分层规则

## 6. 文档样式规范

### 6.1 唯一文档方案

- 使用 **GitHub Flavored Markdown**
- 代码块必须带语言标记
- 双语文档按“中文在前、英文在后”组织

```md
## 中文

### 摘要

...

## English

### Summary

...
```

### 6.2 禁止的文档写法

```md
<!-- ❌ 禁止：只写模糊建议 -->
- maybe do release later

<!-- ❌ 禁止：把 TODO 留在公开 release 文档里 -->
- TODO: add maintainer email

<!-- ✅ 正确：明确状态和下一步 -->
- Remaining blocker: GitHub repository visibility still needs explicit confirmation.
```

## 7. Git 工作流

### 7.1 分支策略

当前仓库默认采用 **轻量 Trunk-Based + 短分支**：

```text
main
 ├── codex/<topic>
 └── docs/<topic>
```

| 规则 | 说明 |
|------|------|
| **主分支** | `main`，始终保持可读、可验证 |
| **功能分支** | 默认 `codex/<short-description>` |
| **文档分支** | 可用 `docs/<short-description>` |
| **合并方式** | 优先 Squash 或直接提交到 `main`（仅在用户明确允许且改动聚焦时） |

### 7.2 分支命名规范

```text
<type>/<short-description>
```

| 类型 | 用途 | 示例 |
|------|------|------|
| `codex/` | AI 执行的功能或规则修改 | `codex/readiness-refinement` |
| `docs/` | 纯文档更新 | `docs/specforge-project-docs` |
| `fix/` | 问题修复 | `fix/release-note-link` |

### 7.3 提交信息规范 (Conventional Commits)

所有 Git 提交信息必须遵循：

```text
<type>(<scope>): <subject>
```

#### 类型 (type)

| 类型 | 说明 | 示例 |
|------|------|------|
| `feat` | 新增能力或文档资产 | `feat(specs): add SpecForge project docs` |
| `fix` | 修复错误或错误结论 | `fix(repo-audit): route no-commit repos to bootstrap` |
| `docs` | 纯文档修订 | `docs(readme): add specs directory` |
| `refactor` | 重构，不改变对外行为 | `refactor(shared): tighten readiness wording` |
| `test` | 增加或调整验证 | `test(validation): add placeholder scan note` |
| `chore` | 工具或仓库维护 | `chore(github): update topics` |

#### 作用域 (scope)

| scope | 对应范围 |
|-------|---------|
| `repo-bootstrap` | `oss-repo-bootstrap` |
| `repo-audit` | `oss-repo-audit` |
| `community-files` | `oss-community-files` |
| `publish-github` | `oss-publish-github` |
| `release-management` | `oss-release-management` |
| `maintainer-ops` | `oss-maintainer-operations` |
| `shared` | `shared/references` / `shared/assets` |
| `docs` | `docs/` |
| `specs` | `specs/` |
| `readme` | `README.md` |

#### 提交信息规则

- `subject` 必须使用英文、祈使语气、小写开头、末尾不加句号
- `body` 解释为什么，而不是重复 diff 做了什么
- 涉及版本或发布的提交必须明确作用域

#### 示例

```text
feat(specs): add SpecForge project governance documents

Document the suite as a SpecForge V3-native project so future project-level
skills can operate on stable product, structure, standards, and roadmap assets.
```

```text
fix(repo-audit): route no-commit repositories to bootstrap

Second-round dogfood showed that local-only repositories were too easy to
misclassify as publication candidates. Tighten readiness-based handoff rules.
```

## 8. 代码注释规范

### 8.1 何时写注释

| 需要注释 | 不需要注释 |
|----------|-----------|
| 解释 handoff 为什么这样设计 | 重复标题含义 |
| 说明为什么某条规则比直觉更严格 | 逐句复述明显文字 |
| 解释外部写入边界 | 对简单列表逐条解释 |
| 说明校验命令为何是门禁 | 对文件名进行口语翻译 |

### 8.2 注释风格

```md
<!-- ✅ 解释 "为什么" -->
This repo treats readiness separately from repository type because multi-repo dogfood proved type alone was not enough.

<!-- ❌ 描述 "做了什么" -->
This section describes readiness states.
```

### 8.3 TODO / FIXME / HACK 标记

```text
TODO: 仅允许出现在内部计划或草稿文档，不允许进入公开 release 文档
FIXME: 只用于明确的已知错误，不得用来代替任务规划
HACK: 必须附上原因与后续替代方向
```

## 9. 错误处理规范

- **本地校验失败**：优先报告具体命令、退出码和失败对象，禁止用笼统表述掩盖
- **GitHub 写入失败**：必须区分登录态、权限、资源不存在、网络波动
- **dogfood 发现问题**：先归类为错误假设、缺失输入、模板过度生成、handoff 不清晰，再决定是否修技能
- **高影响动作前**：必须有写前摘要；失败后要给出可回退依据

## 10. 测试规范

### 10.1 当前仓库测试策略

当前项目以 **结构校验 + 文档一致性校验 + GitHub 状态回读** 为主，不以单元测试为主。

### 10.2 测试优先级

| 优先级 | 内容 |
|--------|------|
| **P0** | 6 个技能的 `quick_validate.py` 通过 |
| **P1** | `git diff --check` 为 0 输出 |
| **P1** | 公开文档无未解析占位符 |
| **P1** | GitHub 声明类结论有 `gh` / API 回读 |
| **P2** | 多仓库 dogfood 验证 |

### 10.3 测试文件与验证命令

- 技能结构验证：`python3 ~/.codex/skills/.system/skill-creator/scripts/quick_validate.py <skill-dir>`
- patch 级检查：`git diff --check`
- 状态检查：`git status --short --branch`
- GitHub 检查：`gh api ...`、`gh release view ...`

## 11. 依赖管理

- 当前仓库默认不新增 repo 级包管理器
- 如需新增脚本依赖，必须先证明现有 `python3`、`git`、`gh`、`rg` 无法完成任务
- 禁止为了单次文档修改引入大型工具链
- 外部工具路径若固定，应在文档中写清楚来源与用途

---

## 12. 环境变量规范

- 当前仓库不应保存任何真实凭据
- GitHub 操作优先通过 `gh auth status` 验证登录态，而不是在仓库内保存 token
- 若后续新增脚本依赖环境变量，必须同步：
  - `README.md`
  - `.env.example`（如果引入）
  - 对应安全说明

安全红线：

- 禁止把 token、私有邮箱、私有仓库链接写入模板或公开文档
- 禁止在 dogfood 报告中泄露目标仓库的敏感路径或私有凭据

## 13. AI 协作协议

### 13.1 写任何内容前必读

| 场景 | 必读文档 |
|------|---------|
| **项目级文档** | `specs/*.md`、`README.md` |
| **技能修改** | 对应 `oss-*/SKILL.md`、`shared/references/*` |
| **发布相关** | `CHANGELOG.md`、`docs/releases/*`、`SECURITY.md` |
| **dogfood 相关** | `docs/dogfood/*` + 目标仓库事实 |

### 13.2 AI 写作与修改规则

| 规则 | 说明 |
|------|------|
| **先读后写** | 没有读当前仓库事实，不允许生成项目级结论 |
| **最小改动** | 只改与当前目标直接相关的文档 |
| **不猜不存在的结构** | 没有 `src/` 就不要写成应用项目结构 |
| **不替用户做高影响决定** | GitHub 可见性、重写 release、删除内容等必须单独确认 |
| **不把 dogfood 结果硬编码成单仓库特例** | 只修共性规则 |
| **无验证不宣称完成** | 完成类表述必须有新鲜证据 |

### 13.3 写完后的检查清单

- 是否与 `specs/` 已有定义矛盾
- 是否与当前 README / CHANGELOG / release 文档矛盾
- 是否把本地可用误写成可公开发布
- 是否残留公开占位符
- 是否需要 `git` / `gh` 回读验证

### 13.4 错题本机制

- **路径**: `specs/AI-错题本.md`
- **记录格式**:

```md
## YYYY-MM-DD
- 场景:
- 错误判断:
- 根因:
- 修正规则:
```

当前阶段该文件可暂不创建，但未来若出现重复错误，必须按此格式落盘。

## 14. 目录依赖方向速查

```text
specs/
  ↓
README.md / docs/
  ↓
oss-*/SKILL.md
  ↘
   shared/references/  ←  shared/assets/
```

禁止的依赖方向：

- `shared/assets/` 反向定义 `shared/references/` 规则
- dogfood 报告直接替代项目治理文档
- release 文档反向定义技能行为

---

## 15. 安全与发布规范

- 任何公开模板都不得包含真实敏感信息
- `SECURITY.md` 必须明确安全上报路径，且不能默认让漏洞走公开 issue
- GitHub private vulnerability reporting 的状态只能通过回读声明
- dogfood 目标仓库默认只读，除非用户明确要求写入
- 对外发布前必须确认：
  - 公开文档无占位符
  - 技能结构校验通过
  - 远端状态回读成功
