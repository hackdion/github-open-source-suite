# 项目开发路线图 (Development Roadmap)

## 0. 策略概览

- **开发策略**: 先把 GitHub 开源发布工作流拆成稳定技能，再用真实仓库 dogfood 收敛边界，最后补齐项目治理资产和后续自动化能力。
- **MVP 目标**: 形成一套可发布、可复用、可在真实仓库上验证的 GitHub 开源发布技能套件。

## 1. 里程碑一：MVP (核心闭环)

> 目标：完成从本地目录到 GitHub 开源发布的最小闭环。

- [x] **Phase 1.1: 基础设施 (Infrastructure)**
  - **模块**: `oss-repo-bootstrap` / `shared`
  - **说明**: 初始化仓库、建立共享规范与模板层、形成技能目录结构。

- [x] **Phase 1.2: 核心业务 (Core Business)**
  - **模块**: `oss-repo-audit` / `oss-community-files` / `oss-publish-github` / `oss-release-management` / `oss-maintainer-operations`
  - **说明**: 打通从 bootstrap、audit、community files、publish、release 到 maintainer operations 的完整阶段流。

## 2. 里程碑二：功能完善 (Feature Complete)

> 目标：让套件不只是“存在”，而是在真实仓库上判断正确、发布正确。

- [x] **Phase 2.1: 当前仓库自身开源发布**
  - **模块**: 根级社区文件、`.github/`、GitHub 仓库设置、release 流程
  - **说明**: 把本仓库自身作为首个公开样本，完成公开发布与 release。

- [x] **Phase 2.2: 第一轮 dogfood**
  - **模块**: `opencode-for-codex-skill`
  - **说明**: 收敛安全上报路径判断、历史分叉与内容差异的区分、release 文档耐久性。

- [x] **Phase 2.3: 第二轮多仓库 dogfood**
  - **模块**: `防停摆-技能` / `独立复核-技能` / `避免重复造轮子`
  - **说明**: 收敛 readiness 模型、早期仓库的社区文件最小集、以及 handoff 顺序。

## 3. 里程碑三：运营与优化 (Ops & Optimization)

> 目标：把当前项目从“已经发布的技能仓库”提升到“可被 SpecForge 连续治理的长期项目”。

- [x] **Phase 3.1: SpecForge 项目资产补齐**
  - **模块**: `specs/`
  - **说明**: 建立 `PROJECT-CONTEXT`、`GUARDRAILS`、产品概述、技术栈、项目结构、开发规范、开发路线图。

- [ ] **Phase 3.2: 自动化验证增强**
  - **模块**: 本地验证脚本 / release 门禁
  - **说明**: 评估是否需要把占位符扫描、公开文档检查、GitHub 回读封装成仓库内稳定脚本。

- [ ] **Phase 3.3: 更大样本泛化验证**
  - **模块**: 新的真实仓库样本
  - **说明**: 增加不同类型项目的 dogfood，验证 readiness 与 handoff 规则是否继续成立。

---

**使用指南**:
1. 当前仓库已经完成 MVP 和两轮主要 dogfood，下一阶段优先做里程碑三。
2. 若后续继续扩展技能边界，先更新 `specs/产品概述.md` 与 `specs/开发路线图.md`，再做实现。
