# 边界守卫 (Guardrails)

> **黄金法则**
>
> 在当前仓库中，任何工作都必须严格区分“项目治理文档”“技能定义”“目标仓库 dogfood”三类对象，禁止混写、越级执行或误改外部仓库。

## 1. 阶段边界

### 1.1 文档阶段

如果当前任务是 SpecForge 项目级文档工作，最终产出只能是：

- `specs/` 下的项目治理文档
- 为保持一致性所需的最小 README / CHANGELOG 引用更新

禁止在这个阶段顺手：

- 改技能逻辑
- 改共享模板内容
- 发 GitHub release
- 修改目标 dogfood 仓库

### 1.2 技能演进阶段

如果当前任务是修技能，产出应限定为：

- `oss-*` 技能文档
- `shared/references/`
- `shared/assets/`
- 对应验证与发布文档

### 1.3 Dogfood 阶段

如果当前任务是 dogfood，默认规则是：

- **只读**
- 不修改目标仓库
- 只记录证据、缺口和 handoff 结论

除非用户明确要求，否则不得在 dogfood 仓库里直接生成或覆盖文件。

## 2. Git 与 GitHub 边界

- 不得把“本地可用”“测试通过”“已有 README/LICENSE”误判为“可公开发布”
- 高影响 GitHub 动作前必须先有本地快照或写前摘要
- 未验证成功前，不得声称“已发布”“已完成”“已修复”
- 不得重写既有 release 或 tag，除非用户明确要求

## 3. 当前仓库的硬约束

- 共享规范优先于临场发挥
- 低风险本地修改可直接执行
- 高影响 GitHub 动作必须读现状、再执行、再回读验证
- 任何目标仓库的用户现有改动都必须保留
- 不得为了补文档而创造并不存在的代码结构、依赖或运行时

## 4. 文档生成约束

- 文档内容必须以当前仓库事实为基础
- 模板中不适用的章节必须明确写“不适用”与原因，不能留空
- 不允许保留未解释的占位符进入公开文档
- 不允许写出无法持续兑现的维护承诺、响应 SLA 或安全承诺

## 5. 验证约束

在当前仓库中，以下结论必须有新鲜证据才能使用：

- 已完成
- 已发布
- 可提交
- 可继续下一阶段
- 已修复

至少要有对应的本地命令输出、git 状态、或 GitHub API / `gh` 回读结果支撑。
