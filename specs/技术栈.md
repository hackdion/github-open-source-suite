# 技术栈选型

## 1. 核心决策 (Key Decisions)

这个项目不是传统 Web 应用，而是一个**文档与技能驱动的工作流仓库**。因此技术选型的原则不是“跑出页面或服务”，而是：

- 优先保证技能定义、共享规范和模板易读、可维护
- 用最少的运行时依赖支撑验证和发布动作
- 让 Git、GitHub CLI 与现有技能生态天然协同

基于这些约束，当前项目采用：

- **Markdown** 作为主要文档载体
- **YAML** 作为技能元数据载体
- **Python 3** 作为轻量验证工具运行时
- **Git + GitHub CLI (`gh`)** 作为版本与发布操作基线

## 2. 前端 (Frontend)

- **框架**: 不适用（当前仓库不是 Web 前端项目）
- **样式方案**: 不适用
- **状态管理**: 不适用

## 3. 后端 (Backend)

- **运行时/语言**: Markdown / YAML 为主；需要脚本辅助时使用 Python 3
- **Web 框架**: 不适用（当前仓库不提供独立 HTTP 服务）
- **API 协议**: 不适用；GitHub 操作通过 `gh` CLI 与 GitHub API 完成

## 4. 数据存储 (Database)

- **主数据库**: 不适用（仓库状态由 Git 与 GitHub 承担）
- **缓存 (可选)**: 不适用

## 5. 基础设施 & 工具 (Infra & Tools)

- **部署**: GitHub 仓库公开发布；通过 GitHub Releases 分发版本
- **包管理**: 仓库级不引入独立包管理器；外部 Python 校验脚本复用现有 `skill-creator` 工具
- **核心工具**:
  - `git`
  - `gh`
  - `python3`
  - `rg`
  - Codex / SpecForge 技能体系
