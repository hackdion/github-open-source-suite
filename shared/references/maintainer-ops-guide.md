# Maintainer Operations Guide

Use this file to define sustainable post-publication operations.

## Core principle

Write rules the maintainer can realistically follow.

## Maintainer operations baseline

Define:

- where questions should go
- how bug reports should be filed
- what a good pull request looks like
- how security issues are reported
- what response expectations are reasonable

## Support boundaries

- Prefer public support channels over direct personal outreach.
- Mark support as best-effort unless the maintainer offers stronger guarantees.
- Separate support from vulnerability reporting.

## Issue triage

Encourage maintainers to sort incoming issues into:

- bug
- feature request
- question or support
- duplicate
- out of scope

If the project is small, lightweight labels and periodic review are enough.

## Pull request intake

State:

- whether unsolicited PRs are welcome
- whether an issue should exist first
- whether docs and tests should be updated with code changes

## Sustainability signals

It is acceptable to say:

- the project is maintained in spare time
- response times may vary
- some issue types may be closed quickly when out of scope

It is better to say this clearly than to imply full-time support.

## Use with the suite

- `oss-maintainer-operations` uses this as the operating baseline.
- `oss-community-files` can reuse its wording for `SUPPORT.md` and `CONTRIBUTING.md`.
